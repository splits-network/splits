Migration Plan - March 9, 2026

Jobs V2 → V3 Migration

Comprehensive migration plan for jobs, job-requirements, job-skills, and saved-jobs resources in ats-service. Includes role-based view design for recruiter, company, candidate, and public consumers.

Critical Correction from Previous Plan

The previous plan concluded that "different roles see different rows but the same columns." This is incorrect. V2 returns identical columns to everyone because it lacked a pattern for role-specific views — not because it is correct. V3 must design proper role-based views with tailored data shapes, joins, and aggregations for each role.

6. Role-Based View Design

Each role gets purpose-built views returning only the data they need, with tailored joins, aggregations, and column exclusions. This is the core improvement over V2.

Recruiter jobs/views/recruiter-board

PropertyValue
URLGET /api/v3/jobs/views/recruiter-board
AuthRequired (recruiter role)
ReplacesGET /v2/jobs when called by recruiter
Frontendapps/portal/src/app/portal/roles/page.tsx (recruiter context)
Columns Returned
id title status fee_percentage guarantee_days location department employment_type commute_types job_level salary_min salary_max source_firm_id company_id activates_at closes_at created_at updated_at
Joins companies id, name, logo_url, industry, headquarters_location
job_skills → skills skill name, is_required (batch-fetched by job IDs)
Aggregations Application count per job (batch-fetched from applications for returned job IDs)
Filters status, job_owner_filter (all|assigned), employment_type, commute_type, job_level, company_id, search
Row Scoping Subscription tier gating: partner tier sees early-access jobs. assigned filter checks job_owner_recruiter_id, company_recruiter_id, applications where recruiter is candidate_recruiter_id, and placements.
Why Separate Recruiters need fee_percentage, guarantee_days, source_firm_id, and subscription-gated visibility. Company users and candidates never see these.

7. Action Endpoints Design

jobs/actions/process-termination

PropertyValue
URLPOST /api/v3/jobs/actions/process-termination
AuthRequired
ReplacesPOST /v2/jobs/termination-decisions
Request Body { recruiter_id: string, decisions: [{ job_id: string, action: "keep" | "pause" | "close" }] }
Response { data: { message: "Termination decisions processed" } }
Side Effects Updates job statuses, nulls recruiter ownership fields. Publishes job.termination_processed event per job.
Status changes use PATCH, not actions. Job status transitions (activate, pause, close, etc.) go through PATCH /v3/jobs/:id with { status: "active" }. The service layer validates the transition. No separate action endpoint per status.

8. Sub-Resources

job-requirements

Stays as a separate V3 resource (not nested under jobs) because it has its own CRUD lifecycle.

V3 RouteMethodNotes
/api/v3/job-requirementsGET, POSTGET requires ?job_id=. No joins needed. Single table.
/api/v3/job-requirements/:idGET, PATCH, DELETEDELETE returns 200 with message (V3 standard). V2 returned 204.
/api/v3/job-requirements/actions/bulk-replacePOSTReplaces V2 PUT. Body: { job_id, requirements[] }. Uses stored procedure.

job-skills

Junction table resource with composite PK. Only 3 of the Core 5 apply.

V3 RouteMethodNotes
/api/v3/job-skillsGET, POSTGET requires ?job_id=. Joins skills table for name/slug (junction table exception).
/api/v3/job-skills/:jobId/:skillIdDELETEComposite key delete.
/api/v3/job-skills/actions/bulk-replacePOSTReplaces V2 PUT. Body: { job_id, skills[] }.
No GET/:id or PATCH/:id — composite PK means no single-row operations by UUID.

saved-jobs

Candidate-only resource. Core CRUD + one view for the enriched list.

V3 RouteMethodNotes
/api/v3/saved-jobsGET, POSTCore GET returns flat rows (no joins). POST is idempotent. Publishes event.
/api/v3/saved-jobs/:idGET, DELETENo PATCH (no editable fields). Hard delete. Publishes event.
/api/v3/saved-jobs/views/enrichedGETJoins jobs → companies. Replaces GET /v2/saved-jobs list for candidate UI.
