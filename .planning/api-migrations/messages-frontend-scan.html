1. Verdict

ZERO FRONTEND FILES NEED MIGRATION

All message API consumers already resolve to V3 endpoints. The default pathPrefix in SplitsApiClient is /api/v3, so every short-path call like /chat/conversations/:id/messages already hits /api/v3/chat/conversations/:id/messages -- which is the registered V3 gateway route.

Why zero migration? The SplitsApiClient (at packages/shared-api-client/src/index.ts line 138) defaults its pathPrefix to /api/v3. All portal and candidate app API clients inherit this default. When frontend code calls client.post('/chat/conversations/123/messages', ...), the client prepends /api/v3, sending the request to /api/v3/chat/conversations/123/messages. This is already the V3 gateway route (registered at services/api-gateway/src/routes/v3/chat.ts lines 31-32). The admin support-chat app uses explicit /api/v3/ full paths, so it is also on V3.

3. URL Construction Mechanics

How Short Paths Become V3

The SplitsApiClient at packages/shared-api-client/src/index.ts (line 163):

client.post('/chat/conversations/123/messages')
→
hasVersionPrefix = false
→
baseUrl + '/api/v3' + '/chat/conversations/123/messages'

Both portal and candidate apps use this client:

App API Client Class Underlying Client Default pathPrefix
PORTAL ApiClient extends AppApiClient SplitsApiClient /api/v3
CANDIDATE ApiClient (wraps SplitsApiClient) SplitsApiClient /api/v3
ADMIN AppApiClient via createAdminClient SplitsApiClient /api/v3
All three app clients default to /api/v3. Short-path calls (those not starting with /api/v) automatically route to V3 endpoints. The admin support-chat uses explicit full /api/v3/... paths via fetch(), which also resolve correctly.

No type changes needed. The frontend message types are a subset of the V3 response. All fields the frontend uses (id, sender_id, body, kind, metadata, created_at) are present in the V3 response. The V3 response includes additional fields (redacted_at, edited_at, etc.) that the frontend can safely ignore since TypeScript types are structural, not nominal.
