Detailed Issues

N+1 Query in LIST Endpoint

service.ts:22-29 — After fetching all recordings, the service loops through unique call_id values and calls getCallParticipants() for each one. This is an N+1 query that will degrade as recording volume grows. The correct fix: the LIST endpoint should be a flat select('*') in core CRUD with no participant checks. Participant-scoped listing belongs in a view.

Broken Pagination

service.ts:33-38 — The total count comes from the database query, but the data array is post-filtered by participant membership. Result: total says 50 but only 12 items returned. total_pages is wrong. Frontend pagination will break.

Wrong URL Pattern for Playback

routes.ts:35 — /api/v3/call-recordings/:id/playback-url is a custom path segment under /:id. Per V3 convention, this should be a view: GET /api/v3/call-recordings/:id/view/playback. It returns a shaped response (signed URL with expiry), not the raw recording row.

Cross-Version Dependency (V3 Imports V2)

v3/recordings/webhook-handler.ts:15 — Imports CallRecordingService from ../../v2/calls/call-recording-service. Also imports IEventPublisher from ../../v2/shared/events. The V3 webhook handler depends on V2 code, blocking V2 removal.

Repository Throws HTTP Errors

repository.ts:50-51,67-68,91 — The V3 repository uses throw Object.assign(new Error(...), { statusCode: ... }). Repositories must never throw HTTP-typed errors. They should return null and let the service decide.

No AccessContextResolver

Neither the V3 service nor repository uses AccessContextResolver. Participant verification is done manually with resolveUserId() + getCallParticipants(). The service should resolve access context from the shared package.

No Typed Errors

All errors use Object.assign(new Error(...), { statusCode }) instead of BadRequestError, NotFoundError, ForbiddenError from @splits-network/shared-fastify.

V3 Route Registry Imports V2

v3/routes.ts:7-8 — The V3 route registry imports IEventPublisher from V2 shared and CallRecordingService from V2. This must be resolved during migration.

The call_id is provided in the request body. The response returns the new recording. This differs from stop, which operates on an existing recording instance.

Role-Based Authorization Analysis

▼
Operation Who Can Access Authorization Method Separate Views Needed?
List recordings Any authenticated user Core CRUD returns flat data. Scoping (by participant) handled via AccessContext in service layer filtering by user's calls. No — same data shape for all roles. Row-level filtering via access context.
Get recording by ID Call participant Service verifies caller is participant on the recording's call. No — same data for all participants.
Playback URL (view) Call participant Service verifies caller is participant. Returns signed URL. No — all participants see same playback URL.
Start recording (action) Call host only Service verifies caller is host via call_participants role check. No — single action, host-gated in service layer.
Stop recording (action) Call host only Same host verification. No — single action, host-gated in service layer.
Webhook LiveKit (no user auth) LiveKit webhook signature verification. No — system-to-system.
No Separate Role-Based Views Needed All roles (host, participant) see the same recording data. The only difference is write permission (host can start/stop), which is enforced in the service layer of each action. Standard pattern: access context + service-level checks.

Dependencies & Risks
  • LiveKit SDK: The start/stop actions require livekit-server-sdk (EgressClient). Ensure this dependency stays in video-service only.
  • Supabase S3 Config: The start action needs S3 credentials for recording upload destination. These env vars must be threaded through to the action service.
  • Frontend depends on call-service V3 migration: The use-call-detail.ts hook currently gets recordings via call-service ?include=recordings. Full frontend migration requires call-service V3 to drop the include pattern.
  • V2 must remain functional: V2 routes stay untouched. Both V2 and V3 coexist until V2 deprecation.
  • Event contract unchanged: The call.recording_ready event payload must stay identical so ai-service and notification-service consumers are unaffected.
