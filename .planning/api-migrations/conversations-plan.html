3. Existing V3 Skeleton Issues

4 files exist at services/chat-service/src/v3/conversations/ but have significant mismatches with V2 behavior. All require full rewrite.

types.ts — Wrong data model
CreateConversationInput uses participant_ids[] for generic multi-party conversations, but V2 is strictly 1:1 with participantUserId + context + representation routing.
repository.ts — Missing enrichment path
findAll does a 2-step query (get participant IDs then filter conversations) but returns flat data. Frontend needs participant names/roles. This belongs in an inbox view, not core CRUD.
service.ts — Wrong create/delete semantics
create() is generic insert, not find-or-create with routing. archive() sets archived: true on chat_conversations, but V2 archives per-participant via archived_at on chat_conversation_participants.
routes.ts — Missing core routes
Only registers GET :id, PATCH :id, DELETE :id. GET list and POST create are missing — comments say "handled by V3 alias in V2 routes." No views/ or actions/ directories exist.

6. Role-Based Authorization

Authorization is participant-based, not role-based. All roles (recruiter, candidate, company_user) use the same endpoints. No role-specific views needed.

Access Model

The service resolves clerkUserId → identityUserId via AccessContextResolver, then checks chat_conversation_participants for membership. You must be a participant to access any conversation endpoint.

Context Access (Start Action Only)

Context access validation differs by role when starting a conversation:
• Candidates: check application.candidate_id
• Recruiters: check application.recruiter_id or job owner/assignee
• Company users: check org membership
The branching is internal to assertContextAccess.

9. Key Risks & Decisions

Existing V3 skeleton has wrong data model
V3 create() uses participant_ids[] for generic multi-party conversations, but V2 is strictly 1:1 with participantUserId + context + representation routing. V3 DELETE archives the conversation table, but V2 archives per-participant.
Decision: Full rewrite of all 4 existing V3 skeleton files. No incremental patching.
V3 aliases in V2 routes.ts create route conflicts
V2 routes.ts registers 10+ routes under /api/v3/* paths. New proper V3 files will conflict. The aliases must be removed when proper V3 handlers are ready.
Decision: Remove V3 aliases from V2 file. This is the ONE exception to "never modify V2" — we're removing V3 paths from V2, not modifying V2 behavior.
DELETE method for mute/archive violates V3 standard
V2 uses POST to mute/archive and DELETE to unmute/unarchive. V3 actions are POST-only. Frontend must change from separate POST/DELETE calls to single POST with boolean body.
Decision: Merge into single POST action with { muted: boolean } / { archived: boolean } body. Frontend change is straightforward.
Core CRUD POST create vs Start action
V3 standard requires Core 5 with POST create, but the real business operation is find-or-create with routing — that's an action. Having both could cause ambiguity.
Decision: Core CRUD POST create = simple insert (internal/service use). Frontend-facing find-or-create with routing = /actions/start. Document clearly.
Messages endpoints overlap with conversations
GET/POST /conversations/:id/messages are in V2 conversation routes but are really messages operations. V3 already has a separate messages resource.
Decision: Leave in V3 messages resource. Do NOT duplicate in conversations migration. Out of scope.
Related resources out of scope
Presence, blocks, reports, attachments, admin moderation are all registered in the same V2 routes.ts file but are independent resources.
Decision: Each gets its own migration plan: /api:plan for blocks, reports, attachments, presence, admin-moderation.
