POST /v2/chat/conversations/:id/messages CRUD POST /v3/chat/conversations/:conversationId/messages DONE Currently implemented as a route in V3 messages/routes.ts. Handles idempotency, real-time events, and domain events. Note: V3 uses direct INSERT rather than the V2 RPC chat_send_message, which may mean V3 is missing the atomic counter/last_message update the RPC provides.
PATCH /v2/chat/messages/:id ACTION POST /v3/messages/:id/actions/redact TODO Admin-only redaction/edit is a side-effect operation (sets redacted_at, publishes event). Should become a V3 action, not a PATCH on core CRUD. The PATCH at /v3/chat/messages/:id currently routes to the V2 handler.
Architecture decision -- messages as sub-resource vs standalone: Messages are accessed as a sub-resource of conversations (/conversations/:id/messages), which is correct because messages always belong to a conversation. However, the admin redaction endpoint operates on messages by ID directly (/messages/:id). In V3, the redaction action should live under /api/v3/messages/:id/actions/redact as a standalone resource action, since the admin operates on the message directly without a conversation context.

10. Risks and Notes

10.1 Critical: V3 send() Missing Business Rules

The V3 MessageService.send() is significantly simpler than V2's ChatServiceV2.sendMessage(). Missing rules:

V2 Rule V3 Status Impact if Missing
Check sender request_state (pending = cannot reply, declined = blocked) MISSING Users could reply before accepting a request
Check recipient request_state (pending = max 1 msg, declined = blocked) MISSING Users could spam pending requests with multiple messages
Check if recipient has archived_at set MISSING Messages sent to archived conversations
Check isBlocked() between sender and recipient MISSING Blocked users could still send messages
Attachment restriction for pending requests MISSING Attachments sent in pending request conversations
Use chat_send_message RPC for atomic insert + counter update DIFFERENT Conversation last_message_at and unread_count may not update atomically

10.2 Messages URL Structure Consideration

Messages are currently registered at /api/v3/chat/conversations/:conversationId/messages, treating them as a sub-resource of conversations. This is correct for list/send operations.

The redact action operates on a message by its own ID. The V3 URL should be: POST /api/v3/messages/:id/actions/redact (standalone resource action). Alternatively, it could be POST /api/v3/chat/conversations/:conversationId/messages/:id/actions/redact, but this is unnecessarily verbose since the admin knows the message ID directly.

10.3 V3 Param Name Discrepancy

The existing V3 routes use :conversationId as the param name (matching the JSON schema conversationIdParamSchema), but the gateway routes use :id. Verify the gateway proxy correctly maps :id in the gateway path to :conversationId in the service path, or align param names.

10.4 Admin Routes Not Yet Migrated

The admin moderation endpoints (/admin/chat/reports, /admin/chat/audit, /admin/chat/metrics) are still V2-only. The message redaction action is a first step, but full admin migration is a separate effort. The admin frontend (chat-moderation-client.tsx) still calls V2 admin paths.
