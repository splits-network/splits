2. Critical: V3 Alias Complication

Frontend currently uses short paths that auto-prefix to /api/v3

Because the SplitsApiClient defaults to /api/v3 base path, all current frontend calls like client.get("/chat/conversations") actually hit /api/v3/chat/conversations.

The V2 routes file registered "fake" V3 aliases at these paths. Now that real V3 endpoints exist at different paths (e.g., /views/list-for-user instead of bare /conversations), the frontend must switch to full V3 paths to hit the proper V3 handlers.

Key insight: The messages sub-resource (/chat/conversations/:id/messages) already has proper V3 routes registered at the same path, so those calls need NO change.

Gateway registers both old alias paths AND new V3 paths
The API gateway at services/api-gateway/src/routes/v3/chat.ts registers both the legacy alias paths (/chat/conversations/:id/accept) and the new V3 action paths (/chat/conversations/:id/actions/accept). This means migration can be incremental -- old paths will continue to work during the transition period. However, the mute and archive endpoints change behavior: the V3 actions use a single POST with boolean body instead of separate POST/DELETE.

7. Body Format Changes (Mute/Archive)

The most significant behavioral change: V2 used separate POST (enable) and DELETE (disable) for mute and archive. V3 uses a single POST with a boolean body.

Mute: V2 Pattern

// Enable mute
client.post(`/chat/conversations/${id}/mute`) // Disable mute
client.delete(`/chat/conversations/${id}/mute`)

Mute: V3 Pattern

// Enable mute
client.post(`/api/v3/chat/conversations/${id}/actions/mute`, { muted: true }) // Disable mute
client.post(`/api/v3/chat/conversations/${id}/actions/mute`, { muted: false })

Archive: V2 Pattern

// Enable archive
client.post(`/chat/conversations/${id}/archive`) // Disable archive
client.delete(`/chat/conversations/${id}/archive`)

Archive: V3 Pattern

// Enable archive
client.post(`/api/v3/chat/conversations/${id}/actions/archive`, { archived: true }) // Disable archive
client.post(`/api/v3/chat/conversations/${id}/actions/archive`, { archived: false })

8. Response Type Changes

V3 list-for-user view returns identical shape to V2 alias
The V3 ListForUserService returns { conversation: { ..., participant_a, participant_b }, participant: { ... } } -- the exact same shape as the V2 alias handler. The frontend ConversationRow type is fully compatible. No type changes needed.
V3 resync action returns identical shape to V2
The V3 resync action returns { data: { conversation, participant, messages } } -- same as V2. The ResyncData type is fully compatible. No type changes needed.
use-candidate-notifications.ts response traversal needs update
The V2 alias returned a flat array where each item has unread_count at the top level. The V3 list-for-user view returns { conversation, participant: { unread_count } }. The candidate notifications hook at line 29 accesses conv.unread_count directly, which will break. Must change to conv.participant?.unread_count || 0.

However, need to verify: the V2 alias normalizeRows() in other consumers suggests the V2 response already uses the nested shape. If the V2 alias returned the same nested shape, this may already work. The normalizeRows function in filter-context.tsx handles both flat and nested shapes. This hook does NOT use normalizeRows and accesses the data directly -- needs verification.

11. Risks and Decisions

RISK: Mute/archive body-change could break if V3 aliases are removed first
The V2 routes file has V3 aliases for mute/archive that accept the old POST/DELETE pattern. If these aliases are removed before the frontend is migrated, the mute/archive functionality will break. Decision: Migrate frontend mute/archive calls BEFORE removing V2 aliases. The gateway registers both old and new paths, so both will work during transition.
RISK: use-candidate-notifications response shape uncertainty
The hook accesses conv.unread_count directly on each conversation item. V3 list-for-user returns { conversation: {...}, participant: { unread_count } }. If the V2 alias already returned this nested shape (which other consumers suggest), this hook was already broken or handles it differently. Decision: Verify actual V2 response shape during migration. Update traversal to use conv.participant?.unread_count || conv.unread_count || 0 for safety.
NOTE: Messages sub-resource is out of scope
GET/POST /chat/conversations/:id/messages is registered at the same path in V3. The auto-prefix already routes these correctly. These calls need NO migration. The messages resource has its own migration plan at .planning/api-migrations/messages-plan.html.
NOTE: Presence, blocks, reports, admin endpoints are out of scope
Several files reference /chat/presence, /chat/blocks, /chat/reports, /admin/chat/*. These are separate resources and not part of the conversations V3 migration.
ADVANTAGE: Gateway supports both old and new paths during transition
The API gateway registers both legacy alias paths (e.g., /chat/conversations/:id/accept) and new V3 action paths (e.g., /chat/conversations/:id/actions/accept). This means migration can be done incrementally per file without breaking anything.
