Key Decision: Presence is NOT a CRUD Resource

Presence has no database table. There are no records to create, update, or delete via REST. The "Core 5" CRUD pattern does not apply. The V3 migration should create:

• 1 View: GET /api/v3/presence/views/batch-status -- batch lookup of user online/idle/offline status from Redis
• 0 Actions: Write path is handled entirely via WebSocket (chat-gateway), not REST
• 0 Core CRUD routes: No database table to CRUD

Data Store

Redis only. No Supabase/Postgres table. All presence data is stored as ephemeral Redis keys with TTLs:

• Key pattern: presence:user:{identityUserId}
• Value: { status: "online"|"idle", lastSeenAt: ISO8601 }
• TTL: 90 seconds (online), 30 seconds (offline)
• Operations: MGET for batch reads, SETEX for writes

Known Bug in V2 POST Endpoints

The V2 POST endpoints (/ping, /offline, /batch) write to presence:user:${clerkUserId} but the GET endpoint reads by identityUserId. Data written via these REST endpoints is never read correctly. The WebSocket-based presence.ping in chat-gateway correctly uses identityUserId.

Architecture Decision: Standalone Resource vs Chat Sub-resource

Recommendation: /api/v3/presence/ as a standalone resource namespace.

Presence is conceptually independent of chat -- it tracks user online/idle/offline status across the entire platform (recruiters, candidates, messaging, calls). It is consumed by recruiter lists, candidate lists, application toolbars, and messages -- not just chat. Nesting it under /chat/presence is a V2 legacy artifact.

The V3 path GET /api/v3/presence/views/batch-status reflects this: presence is a first-class resource, and the batch-status view is its primary (and likely only) read interface.

Why No Core CRUD?

Presence has no database table. There are no presence "records" to list, get by ID, create, update, or delete. The data is ephemeral Redis keys written via WebSocket and read via a single batch GET endpoint. Core 5 CRUD routes would be meaningless.

The V3 presence resource consists of only a single view. This is valid -- not every resource needs CRUD. A view-only resource with no CRUD is appropriate when the resource is read-only from the REST API perspective.

Dependency: Redis Client

The presence repository needs a Redis client, not a Supabase client. The V3 route registration config (RegisterV3Config) currently only passes supabase, eventPublisher, and chatEventPublisher. A Redis client must be added to the config or passed directly to the presence routes registration function.

The existing V2 code creates a Redis instance in registerChatRoutes() from config.redisConfig. The V3 migration should accept a pre-created Redis instance to avoid duplication.

Complexity Assessment

Low complexity. This is a small, focused migration:

• Single view, no CRUD, no actions
• Redis-only (no Supabase schema or migration needed)
• Response shape is unchanged
• Only 2 frontend files need URL updates
• V2 POST endpoints are dead code with zero consumers
• Write path (WebSocket) is already production-stable and unaffected

The main consideration is wiring the Redis client through the V3 registration config, since existing V3 resources only use Supabase.
