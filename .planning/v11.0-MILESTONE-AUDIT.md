## E2E Flow Analysis

### Flow 1: Dashboard Widget → API → Display ✓
- `useUpcomingCalls` → `GET /calls?status=scheduled` → gateway proxy → call-service → response → widget renders
- All wiring confirmed in codebase

### Flow 2: Join Call → Token → Video App ✓
- `useCallToken` → `POST /calls/{id}/token` → gateway → call-service token service → redirect to `video.applicant.network/join/{token}` → video app exchanges token
- Complete chain verified

### Flow 3: Application Detail → Calls Section ✓
- `ApplicationCallsSection` → `GET /calls?entity_type=application&entity_id={id}` → gateway → call-service → `call_entity_links` table → response → renders call cards
- Complete chain verified

### Flow 4: Email Notification URL Routing ✓
- `call.created/cancelled/rescheduled` event → `CallsEventConsumer` → `buildCandidateStatusMap()` → per-participant URL routing → candidates get `candidateWebsiteUrl` links
- Complete chain verified

### Flow 5: In-App Notifications ✗ (Broken)
- `CallInAppNotificationService` defines `notifyCancellation()`, `notifyReschedule()` with `isCandidate` parameter
- **But:** `CallsEventConsumer.handleCallCreated()`, `handleCallCancelled()`, `handleCallRescheduled()` send email only — they never call the in-app service methods
- Only `notifyStartingSoon`, `notifyDecline`, `notifyParticipantJoined` are actually invoked
- No `notifyScheduledCall()` method exists at all for the created event
- **Result:** Candidates receive email but NO bell notification for call lifecycle events

## Data Scoping Concern

`GET /api/v2/calls` in call-service uses `requireUserContext(request)` for auth but does **not** scope the query to the authenticated user's participant records. A candidate calling this endpoint could theoretically see all scheduled calls in the system, not just their own. This is inherited from the portal's broader access model but is a security concern for the candidate app's more restricted trust level.
