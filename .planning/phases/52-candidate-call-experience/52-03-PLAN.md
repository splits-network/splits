Output: Updated notification consumer and in-app service with per-participant URL routing.
</objective>

    1. Update `CallsEventConsumer` constructor to accept `candidateWebsiteUrl`:
       - Add `private candidateWebsiteUrl: string` parameter after `portalUrl`
       - The constructor already has: `emailService`, `inAppService`, `logger`, `portalUrl`, `contactLookup`, `dataLookup`
       - Add `candidateWebsiteUrl` between `portalUrl` and `contactLookup`

    2. Add a private helper method `isCandidateUser(userId: string): Promise<boolean>`:
       - Query the `candidates` table to check if user_id exists: `this.dataLookup` has a Supabase client, OR use `contactLookup`
       - Follow the pattern from `services/notification-service/src/consumers/chat/consumer.ts` line 111-116 which does exactly this:
         ```
         const { data } = await this.repository.supabaseClient
             .from('candidates')
             .select('id')
             .eq('user_id', recipientUserId)
             .maybeSingle();
         return !!data;
         ```
       - Since CallsEventConsumer doesn't have direct supabase access, use `this.dataLookup` which has a supabase property. Check `DataLookupHelper` — it has `this.supabase`. Add a public method `isCandidate(userId: string): Promise<boolean>` to `DataLookupHelper` that queries `candidates` table with `.eq('user_id', userId).maybeSingle()` and returns `!!data`.
       - If adding to DataLookupHelper is cleaner, add it there. Otherwise, use contactLookup which already has supabase access.

    3. Add a private helper method `getJoinUrl(callId: string, isCandidate: boolean): string`:
       - Returns `${this.candidateWebsiteUrl}/portal/calls/${callId}/join` for candidates
       - Returns `${this.portalUrl}/portal/calls/${callId}/join` for portal users
       - Also add `getDetailUrl(callId: string, isCandidate: boolean): string` returning the same pattern but `/portal/calls/${callId}` (without `/join`) — actually candidates don't have a call detail page, so for candidates the detail URL should point to `/portal/dashboard`. Use: `isCandidate ? ${this.candidateWebsiteUrl}/portal/dashboard : ${this.portalUrl}/portal/calls/${callId}`

    4. Update ALL methods that build URLs to use per-participant routing. The places that currently hardcode `this.portalUrl`:

       a. `sendScheduledCallConfirmation()` — line ~475: `joinUrl` and `portalDetailUrl`
          - For each contact, determine if candidate, then pass appropriate URLs
          - The loop already iterates over contacts — add `const isCandidate = await this.isCandidateUser(contact.userId)` inside the loop
          - Pass `joinUrl: this.getJoinUrl(callId, isCandidate)` and `portalUrl: this.getDetailUrl(callId, isCandidate)`

       b. `handleCallRescheduled()` — line ~193: `joinUrl`
          - Same pattern: per-contact URL resolution inside the loop

       c. `handleRecordingReady()` — line ~235: `viewRecordingUrl`
          - Per-contact: candidates get dashboard URL, portal users get call detail URL

       d. `handleStartingSoon()` — line ~291: `joinUrl` in email, and in-app notification
          - Per-contact URL resolution for both email and in-app

       e. `handleReminder()` — line ~349: `joinUrl`
          - Per-contact URL resolution

    5. Update `domain-consumer.ts` to pass `candidateWebsiteUrl` to `CallsEventConsumer`:
       - In the constructor around line 234, the `CallsEventConsumer` is instantiated
       - Currently: `new CallsEventConsumer(notificationService.calls, callInAppService, logger, portalUrl, contactLookup, dataLookup)`
       - Change to: `new CallsEventConsumer(notificationService.calls, callInAppService, logger, portalUrl, candidateWebsiteUrl, contactLookup, dataLookup)`
       - The `candidateWebsiteUrl` is already available in the constructor scope

    IMPORTANT: The for-of loops that iterate contacts now need `await` for the candidate check. This is acceptable — notification sending is already async and not latency-critical. To optimize, you can batch-check all participant userIds upfront:
    ```typescript
    const candidateStatus = new Map<string, boolean>();
    for (const contact of contacts) {
        candidateStatus.set(contact.userId, await this.isCandidateUser(contact.userId));
    }
    ```
    Then use `candidateStatus.get(contact.userId)` in the loop. This avoids N+1 but keeps the code simple.

    AVOID: Don't change the `determineBrand()` method — it determines email template branding (logo/colors), not URL routing. URL routing is separate.
