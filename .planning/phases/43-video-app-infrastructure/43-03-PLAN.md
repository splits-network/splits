    - `call_type` -> `interview_type`
    - `title` -> `title`
    - `scheduled_at` -> `scheduled_at` (default to `new Date().toISOString()` if null)

  <action>
    **call-experience.tsx**: Client component (`'use client'`). This is the main call orchestrator. Accepts `livekitToken: string`, `call: ExchangeResult['call']` props. Manages call state using the shared-video `CallState` type which has 5 states: `'prep' | 'lobby' | 'connecting' | 'in-call' | 'post-call'`.

    Implementation:
    1. Create adapted InterviewContext using `adaptCallToInterviewContext(call)`
    2. Get LiveKit URL from `@splits-network/shared-video` `getLiveKitUrl()` (uses NEXT_PUBLIC_LIVEKIT_URL env var)
    3. Wrap everything in LiveKit `LiveKitRoom` component from `@livekit/components-react`:
       - `serverUrl={livekitUrl}`
       - `token={livekitToken}`
       - `connect={callState === 'connecting' || callState === 'in-call'}`
       - `options={defaultRoomOptions}` from shared-video
    4. State flow handling ALL 5 states:
       - `'prep'`: Initial state. Since the video app uses the join flow (Plan 02) for identity confirmation instead of an in-app prep screen, immediately transition to `'lobby'` on mount via a `useEffect`. The prep state exists in shared-video for the portal's interview flow which has a separate prep step -- the video app skips it because identity confirmation already happened in the join flow.
       - `'lobby'`: Render shared-video `VideoLobby` with the adapted InterviewContext. Include `DeviceSelector`, `AudioLevelMeter`. The lobby's onJoin callback transitions to `'connecting'`.
       - `'connecting'`: Transitional state while LiveKit connection is being established. Show the lobby UI with a loading indicator overlay (DaisyUI `loading loading-spinner`) and "Connecting..." text. Use `useConnectionState()` from `@livekit/components-react` to detect when connection is established, then transition to `'in-call'`. If connection fails after a timeout (10s), show an error and allow retry (back to `'lobby'`).
       - `'in-call'`: Render shared-video `VideoRoom` with `VideoControls`. Include `ReconnectingOverlay` component. Include `CallSidePanel`. Track call duration using `useCallDuration()` hook. The branded header should be hidden or minimized. Leave button in VideoControls transitions to `'post-call'`.
       - `'post-call'`: Render `CallEnded` with call data and duration.
    5. Import `@livekit/components-styles` for base LiveKit styles.
    6. Import `CallState` from `@splits-network/shared-video` -- do NOT redefine the type locally.

    Check the existing VideoLobby and VideoRoom component props in shared-video to ensure correct prop passing. The components were built for the portal interview flow -- pass the adapted InterviewContext where they expect it.

    **call/[callId]/page.tsx**: Client component (`'use client'`). Reads `callId` from params. On mount, reads session data from `sessionStorage.getItem(`call-${callId}`)`. Parse the stored JSON to get `{ livekitToken, call }`. If no session data found (direct URL access without going through /join), redirect to `/error?reason=invalid`. Render `<CallExperience livekitToken={livekitToken} call={call} />`. Clean up sessionStorage after reading.

    Use Next.js 16 `use(params)` pattern for client components to unwrap the params promise.
  </action>
