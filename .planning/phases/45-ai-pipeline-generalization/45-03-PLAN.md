The current setup proxies ALL `/api/v2/calls/*` to call-service. But recording routes (`/api/v2/calls/:id/recording/*`) need to go to video-service since that's where LiveKit integration lives.

Add specific route handlers BEFORE the catch-all `app.all('/api/v2/calls/*', ...)`:

1. **Recording webhook** (no auth):
```typescript
app.post('/api/v2/calls/recording/webhook', async (request, reply) => {
    // No auth headers — LiveKit signature verification
    const correlationId = getCorrelationId(request);
    try {
        const data = await videoService().post(
            '/api/v2/calls/recording/webhook',
            request.body,
            correlationId,
            {},
        );
        return reply.send(data);
    } catch (error: any) {
        request.log.error({ error, correlationId }, 'Failed to proxy call recording webhook');
        return reply.status(error.statusCode || 500).send(error.jsonBody || { error: 'Webhook proxy failed' });
    }
});
```

```typescript
    // ... proxy to videoService() using same switch pattern

```

Also add a route for `/api/v2/calls/:id/recording` (without trailing path) for GET recording status.

Get video-service from ServiceRegistry: `const videoService = () => services.get('video');`

IMPORTANT: These specific routes MUST be registered BEFORE the catch-all `app.all('/api/v2/calls/*', ...)` so Fastify matches them first. Fastify matches routes in registration order for parameterized paths.

Keep these additions minimal — the primary recording flow (start/stop/webhook) lives in video-service. This endpoint is for post-call playback from the portal.
