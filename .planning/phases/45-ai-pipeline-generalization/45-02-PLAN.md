<objective>
Create the generalized AI pipeline in ai-service that transcribes and summarizes call recordings with per-call-type prompts, storing results in the call-owned artifact tables.

Purpose: The existing AI pipeline is interview-specific (reads from `interviews` table, posts summary to `application_notes`). The generalized pipeline reads from `calls`/`call_recordings`, generates per-call-type summaries, and stores results in `call_transcripts` and `call_summaries`. Summaries are never posted to entity note tables (locked decision from CONTEXT.md).

Output: A new `call-pipeline/` module in ai-service with repository, prompt templates, and pipeline service. The domain consumer is extended to listen for `call.recording_ready` events.
</objective>

Create a new `call-pipeline/` directory under `services/ai-service/src/v2/`.

**repository.ts** — Database operations for the call AI pipeline. Mirror TranscriptionRepository patterns but for call tables:

`getCallWithContext(callId)` — Fetch call context for prompt injection:
1. Query `calls` table: id, call_type, title, agenda, created_by, status
2. Query `call_participants` joined with `users` for participant names, roles
3. Query `call_entity_links` to find linked entities
4. For each entity link, fetch context:
   - entity_type='application': get job title, company name, candidate name from applications -> jobs -> companies
   - entity_type='job': get job title, company name from jobs -> companies
   - entity_type='company': get company name from companies
   - entity_type='candidate': get candidate name from candidates
   - entity_type='firm': get firm name from firms
5. Return a `CallContext` interface: `{ call, participants: { name, role }[], entityContext: { type, name, details }[], callType: string }`

`getRecordingSignedUrl(callId)` — Query `call_recordings` for the first recording with status='ready', extract blob_url, generate signed URL from Supabase storage bucket 'interview-recordings' (same bucket, different path prefix `recordings/calls/`).

`updatePipelineStatus(callId, step, status, error?)` — Update pipeline status. Since call tables use separate `call_transcripts` and `call_summaries` tables (unlike interviews which use a column on the interview row), track status by:
- 'transcribing': upsert `call_transcripts` row with transcript_status='processing'
- 'summarizing': upsert `call_summaries` row with summary_status='processing'
- 'complete': update both to 'ready'
- 'failed': update the relevant row to 'failed' with error message

`saveTranscript(input)` — Upload transcript JSON to Supabase storage at `transcripts/calls/{callId}.json`, then upsert `call_transcripts` with storage_url, transcript_status='ready', language.

`saveSummary(callId, summaryData, model)` — Upsert `call_summaries` with:
- summary: JSONB with shape `{ tldr: string, content: string, call_type: string, prompt_version: string, model: string, action_items?: string[], entity_context?: object }`
- summary_status: 'ready'
- model: the OpenAI model used

Constructor takes SupabaseClient and Logger.

**prompt-templates.ts** — Per-call-type prompt templates stored as code constants:

Export a `getPromptForCallType(callType, context)` function that returns `{ systemPrompt, userPrompt }`.

`INTERVIEW_PROMPT`: Extend the existing v9.0 interview prompt with improvements:
- System: "You are summarizing a video interview for a recruiter."
- Include entity context: job title, company name, candidate name (from context)
- Start with a one-line TL;DR (locked decision)
- Sections: Key Discussion Points, Strengths, Concerns, Overall Impression
- Keep existing rules: 200-300 words, bullet points, no hire/no-hire recommendations
- prompt_version: "interview-v2"

`CLIENT_MEETING_PROMPT`: New recruiting call prompt:
- System: "You are summarizing a recruiter-company business meeting."
- Include entity context: company name, job title if linked, participant names/roles
- Start with a one-line TL;DR (e.g., "Agreed to 3 new job postings, follow-up next Tuesday")
- Sections:
  - Action Items (with owners if identifiable): next steps and commitments
  - Key Decisions: what was agreed upon
  - Discussion Summary: 3-5 bullet points of main topics
  - Follow-ups: items needing future attention
- 200-400 words, bullet points
- prompt_version: "client_meeting-v1"

Both prompts truncate transcript to 12000 chars and job description to 2000 chars (matching existing limits).

Export `PROMPT_VERSIONS` constant mapping call_type -> version string for tracking.

**service.ts** — Orchestrate the call AI pipeline. Mirror TranscriptionPipelineService but for calls:

`CallPipelineService` class with constructor(repository: CallPipelineRepository, eventPublisher, logger).

`processRecording(event: { call_id, recording_url, duration_seconds, file_size_bytes })`:
1. Set pipeline status: transcribing
2. Get call context via repository.getCallWithContext(call_id) — for prompt injection
3. Get signed URL via repository.getRecordingSignedUrl(call_id)
4. Download recording (reuse same downloadRecording pattern with 5-min timeout)
5. If file > 25MB, extract audio via FFmpeg (reuse same extractAudioWithFfmpeg pattern)
6. Call Whisper API (reuse same callWhisperApi pattern with retry)
7. Map segments to speakers — for calls, use participant names from context. Since calls don't have interviewer/candidate roles, map by participant order: first participant = first speaker, alternate on gaps > 1s. Use actual participant names.
8. Save transcript via repository.saveTranscript()
9. Set pipeline status: summarizing
10. Get prompt for call type via getPromptForCallType(context.callType, context)
11. Call OpenAI chat completion with the prompt (reuse SummaryService pattern — BUT since the prompt format is now structured differently, instantiate the OpenAI call directly in this service rather than reusing SummaryService which is interview-specific)
12. Parse the response. The summary output is markdown text from the model. Build the JSONB object: `{ tldr: extractTldr(response), content: response, call_type: context.callType, prompt_version: PROMPT_VERSIONS[context.callType], model: modelUsed, action_items: extractActionItems(response) }`
  - `extractTldr(markdown)`: Look for the first line that starts with "**TL;DR:**" or "TL;DR:" and extract the text. If not found, use the first sentence.
  - `extractActionItems(markdown)`: Look for "## Action Items" section, extract bullet points as string array. Return undefined if no section found.
13. Save summary via repository.saveSummary()
14. Set pipeline status: complete
15. Publish `call.summary_ready` event with { call_id, call_type, has_action_items }
16. On error: set pipeline status to 'failed' with error message. Do NOT rethrow (same pattern as interview pipeline).

Use same retry strategy as existing pipeline: 2 retries with exponential backoff for Whisper and OpenAI calls. Use `process.env.OPENAI_API_KEY` and `process.env.OPENAI_SUMMARY_MODEL || 'gpt-4o-mini'`.
