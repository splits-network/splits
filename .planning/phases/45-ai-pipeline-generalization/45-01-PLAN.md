<objective>
Wire up call recording infrastructure in video-service so that completed call recordings are stored in the `call_recordings` table and a `call.recording_ready` event is published for the AI pipeline.

Purpose: The existing recording infrastructure only handles interviews (stored on `interviews` table, publishes `interview.recording_ready`). The generalized call system has `call_recordings`, `call_transcripts`, and `call_summaries` tables but no recording pipeline. This plan creates the recording flow for calls.

Output: A new `calls/` module in video-service mirroring the interview recording pattern but operating on call tables.
</objective>

**call-recording-service.ts**: Mirror `RecordingService` but for calls. Key differences:
- `startRecording(callId)` — looks up call (not interview), verifies status is 'active', creates `call_recordings` row with status 'pending', starts LiveKit egress with filepath `recordings/calls/${callId}/${recordingId}.mp4`, updates recording status to 'recording' with egress_id
- `stopRecording(callId)` — finds active recording in `call_recordings` by call_id where recording_status='recording', stops egress, sets status to 'processing'
- `handleEgressComplete(egressId, fileUrl, duration, fileSize)` — finds recording by egress_id, updates to 'ready' with blob_url, duration, file_size. Returns `{ callId, recordingId }`
- `handleEgressFailed(egressId, error)` — finds recording by egress_id, sets status to 'failed'
- Constructor takes CallRecordingRepository, LiveKit credentials, S3 config (same pattern as interview RecordingService)

**call-recording-webhook.ts**: Mirror `recording-webhook.ts` but for calls.
- POST `/api/v2/calls/recording/webhook` — no auth, LiveKit signature verified via WebhookReceiver
- On `EGRESS_COMPLETE`: call `callRecordingService.handleEgressComplete()`, then publish `call.recording_ready` event with payload: `{ call_id, recording_url, duration_seconds, file_size_bytes }`
- On `EGRESS_FAILED`: call `callRecordingService.handleEgressFailed()`
- Always return 200 to prevent retry storms (same pattern as interview webhook)

**call-recording-routes.ts**: Mirror `recording-routes.ts` for calls.
- POST `/api/v2/calls/:id/recording/start` — authenticated, verify caller is a call participant (host role), call `startRecording(callId)`
- POST `/api/v2/calls/:id/recording/stop` — authenticated, verify caller is host, call `stopRecording(callId)`
- GET `/api/v2/calls/:id/recording/playback-url` — authenticated, verify caller is participant, generate signed URL from `call_recordings` blob_url using the existing `generateSignedUrl` helper
- For participant verification: query `call_participants` for matching user_id. Unlike interviews which require 'interviewer' role, calls require 'host' role for start/stop but any participant role for playback.
