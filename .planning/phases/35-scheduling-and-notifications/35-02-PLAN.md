  <name>Task 1: Add calendar event update, delete, and ICS generation</name>
  <files>services/integration-service/src/v2/calendar/service.ts, services/integration-service/src/v2/calendar/routes.ts, services/integration-service/src/v2/calendar/google-client.ts, services/integration-service/src/v2/calendar/microsoft-client.ts, services/integration-service/src/v2/calendar/ics-generator.ts</files>
  <action>
1. **In google-client.ts** — add methods:
   - `updateEvent(token, calendarId, eventId, params)` — PATCH to Google Calendar API v3 `/calendars/{calId}/events/{eventId}`
   - `deleteEvent(token, calendarId, eventId)` — DELETE to Google Calendar API v3

2. **In microsoft-client.ts** — add methods:
   - `updateEvent(token, calendarId, eventId, params)` — PATCH to MS Graph `/me/calendars/{calId}/events/{eventId}`
   - `deleteEvent(token, calendarId, eventId)` — DELETE to MS Graph

3. **In service.ts** — add methods:
   - `updateEvent(connectionId, clerkUserId, calendarId, eventId, params: UpdateCalendarEventParams)` — dispatches to Google or Microsoft client. `UpdateCalendarEventParams` includes optional: summary, description, startDateTime, endDateTime, timeZone, attendeeEmails
   - `deleteEvent(connectionId, clerkUserId, calendarId, eventId)` — dispatches to Google or Microsoft client

4. **In routes.ts** — add routes:
   - `PATCH /api/v2/integrations/calendar/:connectionId/events/:eventId` — calls `service.updateEvent`
   - `DELETE /api/v2/integrations/calendar/:connectionId/events/:eventId` — calls `service.deleteEvent`
   Both require `calendar_id` in body/query.

5. **Create ics-generator.ts** — utility module:
   - Export `generateICS(params: { summary, description, startDateTime, endDateTime, timeZone, attendeeEmails, location?, meetingLink? }): string`
   - Returns RFC 5545 compliant VCALENDAR string with VEVENT
   - Include DTSTART, DTEND with TZID, SUMMARY, DESCRIPTION, ATTENDEE, ORGANIZER, LOCATION, URL (for meeting link)
   - Use `crypto.randomUUID()` for UID
   - No external dependencies — pure string template

Follow existing patterns in the calendar service: authorize connection, get valid token, dispatch to provider client, normalize response.
  </action>

  <done>Calendar events can be updated and deleted via API for both Google and Microsoft providers. ICS files can be generated as strings.</done>

<task type="auto">
  <name>Task 2: Add calendar webhook subscription service</name>
  <files>services/integration-service/src/v2/calendar/webhook-service.ts, services/integration-service/src/v2/calendar/routes.ts</files>
  <action>
Create a webhook service that manages calendar change subscriptions for sync-back:

1. **Create webhook-service.ts:**
   - `CalendarWebhookService` class with constructor taking `connectionRepo`, `tokenRefresh`, `logger`, `webhookBaseUrl` (from env `INTEGRATION_WEBHOOK_URL`)
   - `subscribeToChanges(connectionId, clerkUserId, calendarId)`:
     - For Google: POST to `/calendars/{calId}/events/watch` with webhook URL `{webhookBaseUrl}/api/v2/integrations/webhooks/google`, generate channel ID as UUID, set TTL 7 days
     - For Microsoft: POST to MS Graph `/subscriptions` with changeType `updated,deleted`, resource `/me/calendars/{calId}/events`, notificationUrl `{webhookBaseUrl}/api/v2/integrations/webhooks/microsoft`, expirationDateTime 3 days (MS Graph max)
     - Store subscription details in connection metadata (JSONB field) for renewal tracking
   - `handleGoogleWebhook(headers)` — parse X-Goog-Channel-ID, X-Goog-Resource-State headers. Return `{connectionId, calendarId, changeType}`
   - `handleMicrosoftWebhook(body)` — parse subscription notification. Return `{connectionId, calendarId, changeType, resourceId}`

2. **In routes.ts** — add webhook receiver routes:
   - `POST /api/v2/integrations/webhooks/google` — NO AUTH (Google sends this). Calls `webhookService.handleGoogleWebhook`, publishes `calendar.event_changed` RabbitMQ event with payload `{connection_id, calendar_id, change_type}`. Return 200.
   - `POST /api/v2/integrations/webhooks/microsoft` — NO AUTH. Handles validation token if query `validationToken` present (return it as text/plain). Otherwise calls `webhookService.handleMicrosoftWebhook`, publishes same event. Return 202.

Note: The actual sync-back processing (fetching changed events and updating interview records) will be handled by the video-service consumer in a later plan. These webhooks just publish events.

Keep the webhook service focused: receive webhook -> publish domain event. Don't process the changes here.
  </action>
