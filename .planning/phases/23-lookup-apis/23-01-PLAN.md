  <action>
Create three files mirroring the skills module (`services/ats-service/src/v2/skills/`) with these substitutions:

**repository.ts — `PerkRepository`:**
- Constructor takes `supabaseUrl` and `supabaseKey`, creates Supabase client (same as SkillRepository)
- `getSupabase()` method exposing the client
- `search(query, limit=20)` — query the `perks` table with `.ilike('name', '%query%')`, order by name, limit. Min 1 char (return empty if less). Use limit default of 20 (more results than skills since perks are fewer/broader)
- `getById(id)` — `.eq('id', id).maybeSingle()`, throw if not found
- `getBySlug(slug)` — `.eq('slug', slug).maybeSingle()`, return null if not found
- `create(name, slug, createdBy?)` — insert into `perks`, select and return single

**service.ts — `PerkService`:**
- Constructor takes `PerkRepository` and optional `SupabaseClient` (for AccessContextResolver, same pattern as SkillService)
- `searchPerks(query, limit=20)` — delegates to repository.search
- `findOrCreate(name, clerkUserId?)` — trim name, generate slug using same `generateSlug` function (copy from skills service: lowercase, strip non-alphanumeric except +#.- , collapse whitespace), check existing by slug, return if found. If not found, resolve clerkUserId to UUID via AccessContextResolver (same pattern as SkillService), then create
- `getPerk(id)` — delegates to repository.getById

**routes.ts — `registerPerkRoutes`:**
- `GET /api/v2/perks` — requireUserContext, parse `q` and `limit` from query. If `q` provided, call `service.searchPerks(q, limit)`. Return `{ data }`. No list-all endpoint needed (unlike skills, perks only need search for typeahead)
- `POST /api/v2/perks` — requireUserContext, parse `{ name }` from body, call `service.findOrCreate(name, clerkUserId)`. If the perk already existed (slug match), return 200. If newly created, return 201. Both return `{ data }`. To distinguish: have findOrCreate return `{ perk, created: boolean }` internally, then routes uses that to pick status code
- Error handling: same try/catch pattern as skills routes, return `{ error: { message } }`
