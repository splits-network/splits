  <action>
    Create the interview page route at `apps/portal/src/app/portal/interview/[id]/`.

    **layout.tsx** — Minimal layout for interview pages:
    - No sidebar, no header, no navigation — full screen for the call
    - Just render `{children}` in a full-screen container
    - Import `@livekit/components-styles` CSS here (or in the page)

    **page.tsx** — Server component:
    - Extract `id` from params (interview ID)
    - Render `<InterviewClient interviewId={id} />` as client component
    - Metadata: `title: "Interview | Splits Network"`

    **interview-client.tsx** — Client component with state machine (this is the core):
    - `'use client'` directive
    - Props: `{ interviewId: string }`
    - State: `callState: CallState` (from shared-video types), starts at `'lobby'`
    - State: `livekitToken: string | null`, `userChoices: LocalUserChoices | null`, `interviewContext: InterviewContext | null`

    On mount:
    - Use `useInterviewToken` hook from shared-video
    - Call `fetchAuthenticatedToken(interviewId)` to get token and interview context
    - Store `interviewContext` in state (for lobby display)
    - Do NOT store the LiveKit JWT yet — that's fetched fresh when joining

    State machine rendering (following RESEARCH.md Pattern 1):
    ```
    'lobby' -> <VideoLobby interviewContext={interviewContext} onJoin={handleJoin} participantPresence={presence} />
    'connecting' | 'in-call' -> <LiveKitRoom token={livekitToken} serverUrl={getLiveKitUrl()} connect={true} audio={userChoices.audioEnabled} video={userChoices.videoEnabled} onConnected={() => setCallState('in-call')} onDisconnected={() => setCallState('post-call')}><VideoRoom interviewContext={interviewContext} ... /></LiveKitRoom>
    'post-call' -> <PostCallSummary duration={duration} isCandidate={false} onClose={handleClose} />
    ```

    CRITICAL anti-patterns to avoid:
    - Do NOT mount `<LiveKitRoom>` during lobby state (RESEARCH.md anti-pattern)
    - Do NOT put token in a dependency that causes re-render (RESEARCH.md Pitfall 2)
    - Generate token ONCE, store in ref or stable state

    `handleJoin` callback:
    - Receives `LocalUserChoices` from lobby
    - Fetches a fresh LiveKit token (or reuses the one from initial fetch)
    - Sets `livekitToken`, `userChoices`, transitions to `'connecting'`

    `handleClose` callback:
    - `window.close()` to close the new tab

    Error handling:
    - If token fetch fails: show error screen with message and "Close" button
    - If interview is expired/cancelled (410 from API): show "Interview no longer available" message
    - If browser doesn't support getUserMedia: show unsupported browser message with list of supported browsers

    Use `useCallDuration` hook from shared-video to track call duration (start on 'in-call', stop on 'post-call').

    Participant presence in lobby: For Phase 34, use the approach from RESEARCH.md Open Question 3 option (a) — connect to LiveKit room with audio=false video=false to detect if the other participant is already in. However, this adds complexity. SIMPLER ALTERNATIVE for v1: skip real-time presence detection. Instead, just show the other participant's name from interviewContext (e.g., "You'll be meeting with {name}"). Real-time "is waiting" can be enhanced later. Use this simpler approach.
