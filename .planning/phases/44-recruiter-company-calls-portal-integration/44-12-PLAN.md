**Look at how existing in-app notifications work** — check the notification-service for how it creates in-app notification records (likely a `notifications` table or similar). Follow the existing pattern.

**Create `in-app-service.ts`:**
- Class `CallInAppNotificationService`
- Methods that create notification records for each participant:

1. **`notifyInstantCall(event)`** — creates urgent notification:
   - Type: `call.instant`
   - Title: "{caller_name} is calling you"
   - Body: entity context if available
   - Actions: accept_url (join URL), decline_url (POST /api/v2/calls/:id/decline)
   - Priority: high (for toast display)

2. **`notifyStartingSoon(event)`** — creates notification:
   - Type: `call.starting_soon`
   - Title: "Call starting in 5 minutes"
   - Body: "{title} with {participant_names}"
   - Actions: join_url
   - Priority: high (for toast)

3. **`notifyParticipantJoined(event)`** — creates notification:
   - Type: `call.participant_joined`
   - Title: "{name} joined the call"
   - Body: call title
   - Priority: medium (toast)

4. **`notifyDecline(event)`** — creates notification for caller:
   - Type: `call.declined`
   - Title: "{name} declined your call"
   - Priority: medium (toast + bell)

5. **`notifyRecordingReady(event)`** — creates notification:
   - Type: `call.recording_ready`
   - Title: "Recording ready"
   - Body: call title, link to detail page
   - Priority: low (bell only)

6. **`notifyCancellation(event)`** / **`notifyReschedule(event)`** — bell notifications (not toast)

**Update consumer.ts:**
- Wire new events to in-app-service methods alongside email methods:
  - `call.created` (instant) → notifyInstantCall (in addition to email)
  - `call.starting_soon` → notifyStartingSoon
  - `call.participant.joined` → notifyParticipantJoined
  - `call.declined` → notifyDecline
  - `call.recording.ready` → notifyRecordingReady (in addition to email)
  - `call.cancelled` → notifyCancellation (in addition to email)
  - `call.rescheduled` → notifyReschedule (in addition to email)

**Add decline route to call-service** if not exists:
- `POST /api/v2/calls/:id/decline` — marks the requesting user's participant record as declined, publishes `call.declined` event with decliner info
