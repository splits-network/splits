- `removeCallTags(callId: string, tagSlugs: string[])` вЂ” delete from call_tag_links
- `getCallTags(callId: string): CallTagLink[]` вЂ” fetch tags for a call
- `listTags(): CallTag[]` вЂ” fetch all available tags from call_tags lookup

**In repository.ts**, extend:
1. **`createCall`** вЂ” accept and insert `agenda`, `duration_minutes_planned`, `pre_call_notes` fields
2. **`updateCall`** вЂ” accept `agenda`, `needs_follow_up` updates
3. **`listCalls`** вЂ” add support for:
   - `tag` filter: join with `call_tag_links` WHERE `tag_slug = ?`
   - `needs_follow_up` filter: WHERE `needs_follow_up = true`
   - `search` filter: search across participant user names and entity link names. Use a subquery approach:
     - Join `call_participants` -> `users` to search first_name/last_name
     - Join `call_entity_links` -> relevant entity tables for entity name search
     - Use ILIKE with `%search%` pattern
   - Make sure the count query also applies these filters for correct pagination
4. **`cancelCall`** вЂ” new method accepting `cancelledBy: string` and optional `cancelReason: string`, sets `cancelled_by` and `cancel_reason` columns alongside status update
5. **`getCallStats(userId: string)`** вЂ” new method returning:
   - `upcoming_count`: calls with status='scheduled' where user is participant and scheduled_at > now
   - `this_week_count`: completed calls this week where user is participant
   - `this_month_count`: completed calls this month where user is participant
   - `avg_duration_minutes`: average duration_minutes of completed calls where user is participant
   - `needs_follow_up_count`: calls with needs_follow_up=true where user is participant

   Use a single query with conditional aggregation for efficiency.

Ensure all new methods use parameterized queries (not string interpolation) for SQL injection prevention.
