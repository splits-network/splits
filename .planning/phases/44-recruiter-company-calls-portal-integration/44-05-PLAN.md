  <files>

  <action>
Look at the existing interview calendar event methods in `service.ts` as the pattern. Extend with call-specific methods:

1. **`createCallCalendarEvent(params)`** — accepts:
   - `callId: string`
   - `title: string` (call title or "Call with {participant_names}")
   - `scheduledAt: string` (ISO datetime)
   - `durationMinutes: number` (planned duration, default 30)
   - `agenda: string | null`
   - `participants: { userId: string; email: string; name: string }[]`
   - `joinUrl: string` (magic link URL for the video app)

   For each participant with a connected Google Calendar:
   - Create a calendar event with:
     - Summary: title
     - Start/end time based on scheduledAt + durationMinutes
     - Description: agenda text + "Join video call: {joinUrl}"
     - Location: joinUrl (so calendar apps show "Join" button)
     - Attendees: all participant emails
   - Store the Google Calendar event ID on a `call_calendar_events` mapping table (or in-memory — use whichever pattern interviews use)
   - Skip participants without connected Google Calendar (falls back to manual join)

2. **`updateCallCalendarEvent(callId, newScheduledAt, newDurationMinutes?)`** — look up stored calendar event IDs, update start/end time for each participant's event

3. **`deleteCallCalendarEvent(callId)`** — look up stored calendar event IDs, delete each

**Add routes** for internal service-to-service calls if the existing pattern uses HTTP. If interviews use RabbitMQ events for calendar sync, create a consumer instead:
- Check how interview scheduling triggers calendar event creation
- Follow the SAME pattern (likely a consumer in integration-service listening to interview events)
- Add `call.created`, `call.rescheduled`, `call.cancelled` event handlers that call the above methods

**If a `call_calendar_events` tracking table is needed** (to store Google Calendar event IDs for updates/deletes), create a migration:
- `services/call-service/migrations/20260308000003_call_calendar_events.sql`
- Table: `call_calendar_events` with `call_id`, `user_id`, `google_event_id`, `created_at`
- Or use the pattern interviews already use for tracking calendar event IDs

The CONTEXT.md specifies slot-based scheduling from Google Calendar availability. Add an endpoint to fetch availability for call scheduling:

1. **In integration-service**, add route `GET /api/v2/calendar/availability`:
   - Query params: `user_ids` (comma-separated), `date_from`, `date_to`
   - For each user with connected Google Calendar: query freebusy API
   - Return: `{ data: { user_id: string, busy_slots: { start: string, end: string }[] }[] }`
   - If user has no calendar connected: return empty busy_slots for that user (UI will show free-form picker)

2. **In gateway integrations route**, add proxy for the availability endpoint if not already proxied by the wildcard pattern. Check existing integrations.ts — if it uses a wildcard catch-all for `/api/v2/integrations/*`, this may already be covered. If not, add explicit route.

The portal scheduling UI (Plan 44-07) will call this endpoint to show available time slots when both participants have Google Calendar connected, falling back to a free-form date/time picker otherwise.
