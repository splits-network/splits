**types.ts changes:**
- Add `recording_enabled: boolean` and `ai_analysis_enabled: boolean` to the `Call` interface
- Add `recording_enabled?: boolean` and `ai_analysis_enabled?: boolean` to `CreateCallInput`
- Remove `requires_recording_consent` and `supports_ai_summary` from the `CallType` interface
- Remove `recording_consent_required?: boolean` from `CallDetail` interface

**repository.ts changes:**
- In `createCall()`, include `recording_enabled: input.recording_enabled ?? false` and `ai_analysis_enabled: input.ai_analysis_enabled ?? false` in the insert object
- In `getCallDetail()`, remove the call_types lookup block that reads `requires_recording_consent` and sets `recording_consent_required` on the detail object. The call object itself now carries `recording_enabled` directly.
- Add a new method `getCreatorTier(userId: string)` that queries subscriptions + plans to get the creator's current plan tier:
  ```
  async getCreatorTier(userId: string): Promise<'starter' | 'pro' | 'partner'> {
      const { data: sub } = await this.supabase
          .from('subscriptions')
          .select('plan:plans(tier)')
          .eq('user_id', userId)
          .eq('status', 'active')
          .order('created_at', { ascending: false })
          .limit(1)
          .maybeSingle();
      return (sub?.plan as any)?.tier || 'starter';
  }
  ```

**service.ts changes:**
- In `createCall()`, after resolving the user ID and before creating the call record, add tier validation:
  - If `input.ai_analysis_enabled` is truthy, call `this.repository.getCreatorTier(resolvedUserId)` to get the tier
  - If tier is not `'partner'`, throw a 400 error: `'AI analysis requires a Partner subscription'`
- Include `recording_enabled` and `ai_analysis_enabled` in the `call.created` event payload

**routes.ts changes:**
- In the POST /api/v2/calls handler, forward `recording_enabled: body.recording_enabled` and `ai_analysis_enabled: body.ai_analysis_enabled` in the input object passed to `service.createCall()`
