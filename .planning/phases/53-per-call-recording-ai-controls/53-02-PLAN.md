<objective>
Add subscription tier enforcement to the AI pipeline so it respects per-call recording/AI flags and the creator's current subscription tier before processing recordings.

Purpose: The AI pipeline currently processes all recordings unconditionally. It must now check the call's recording_enabled and ai_analysis_enabled flags, plus the creator's current tier, to decide what processing to perform. This enforces the tier model: Free gets nothing, Pro gets transcription, Partner gets full pipeline.
Output: Modified pipeline repository with tier lookup, modified pipeline service with conditional processing.
</objective>

Add two new methods to `CallPipelineRepository`:

1. `getCallFlags(callId: string)` — returns `{ recording_enabled: boolean, transcription_enabled: boolean, ai_analysis_enabled: boolean, created_by: string }` by querying the calls table. Returns null if call not found.

2. `getCreatorTier(userId: string)` — queries subscriptions joined with plans to get the creator's current plan tier. Returns `'starter' | 'pro' | 'partner'`. Logic:
   - Query `subscriptions` table with `.select('plan:plans(tier)')` where `user_id = userId` and `status = 'active'`, ordered by `created_at desc`, limit 1, `.maybeSingle()`
   - Return `(sub?.plan as any)?.tier || 'starter'` (default to starter if no active subscription)

Both methods use the existing `this.supabase` service-role client (no HTTP calls between services).

<task type="auto">
  <name>Task 2: Add tier-gated decision tree to pipeline service</name>
  <files>services/ai-service/src/v2/call-pipeline/service.ts</files>
  <action>
Modify the `processRecording` method to add tier checks at the very start, before any processing:

1. At the top of `processRecording` (inside the try block, before the existing Step 1), add a new step:
   - Call `this.repository.getCallFlags(call_id)` to get `recording_enabled`, `transcription_enabled`, `ai_analysis_enabled`, and `created_by`
   - If call not found, log warning and return early (no processing)
   - If `recording_enabled` is false, log info "Recording not enabled for call, skipping pipeline" and return early

2. After confirming recording is enabled, get the tier:
   - Wrap the `getCreatorTier` call in a try/catch. If it fails (billing DB unavailable), log error "Failed to check creator tier, skipping for retry" and return early (don't lose recording, don't throw — the event can be retried later)
   - If tier is `'starter'`, log info "Starter tier — skipping transcription and summarization" and return early

3. For the transcription step (existing Step 3-4), gate it:
   - Only proceed with transcription if `transcription_enabled === true`
   - If transcription_enabled is false, skip transcription and summarization entirely (AI analysis depends on transcript)
   - When skipping, log: "Transcription not enabled for call, skipping"

4. For the summarization step (existing Step 5-7), gate it:
   - Only proceed with summarization if `tier === 'partner'` AND `ai_analysis_enabled === true`
   - Otherwise, skip summarization. Update the pipeline status to 'complete' after saving transcript, and skip the summary steps entirely.
   - When skipping summarization, log: "Skipping AI summarization (tier={tier}, ai_analysis_enabled={flag})"

5. Adjust the final Step 7 (updatePipelineStatus to 'complete') — it should still fire whether or not summarization ran. Move it to a unified location after the conditional summarization block.

6. The event publish in Step 8 should still fire for transcription-only completions.

The decision tree in summary:
```
if (!recording_enabled) return
if (tier === 'starter') return
if (transcription_enabled) transcribe()
if (tier === 'partner' && ai_analysis_enabled && transcription_enabled) summarize()
complete()
```
  </action>

- Billing DB failure causes graceful skip (no crash, no lost recording)
  </done>
