<objective>
Wire Google Calendar auto-creation when a scheduled call is created from the portal.

Purpose: The calendar API (POST /api/v2/integrations/calendar/calls) exists and works but nothing triggers it. Per decision [44-05], calendar operations use HTTP routes (not RabbitMQ consumers). The cleanest approach is the portal frontend calling the calendar API after call creation succeeds, since the frontend already has all participant data and follows the "frontend calls api-gateway only" rule.

Output: After creating a scheduled call, the portal automatically fires a background request to create Google Calendar events for each participant who has a connected calendar.
</objective>

In `call-creation-modal.tsx`, after a **scheduled** call is successfully created, fire a background calendar event creation.

**In handleSubmit, after `const call = await createCall(payload);`:**

Add this block for scheduled calls only (NOT instant calls):
```typescript
if (mode === 'scheduled' && schedule) {
    // Fire-and-forget: create calendar events for participants with connected calendars
    createCalendarEvents({
        callId: call.id,
        title: title || `Call with ${otherParticipants.map(p => p.first_name).join(', ')}`,
        scheduledAt: schedule.dateTime,
        durationMinutes: duration,
        agenda: agenda || undefined,
        participants: participants.filter(p => !p.user_id.startsWith('email:')),
    }).catch(() => {
        // Silently ignore — calendar is best-effort
    });
}
```

**Key rules:**
- Do NOT await the calendar call — it's fire-and-forget. The `onSuccess` callback and `onClose` should NOT wait for calendar.
- Only trigger for `mode === 'scheduled'` — instant calls have no scheduled_at and no calendar event to create.
- Filter out email-only participants (user_id starts with "email:") since they won't have calendar connections.
- Import `createCalendarEvents` from the destructured `useCreateCall()` return.
