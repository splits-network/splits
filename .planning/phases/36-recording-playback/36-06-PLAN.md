**Portal interview-client.tsx** -- Integrate recording:
1. Import `useRecordingState` from `@splits-network/shared-video`
2. After fetching interview context, determine if recording is enabled from the interview data
3. Determine if current user is the interviewer (compare user ID against participant roles)
4. Initialize `useRecordingState(interviewId, API_BASE, getToken, isInterviewer)`
5. In the lobby state:
   - Pass `recordingEnabled={interviewContext.recording_enabled}` to `VideoLobby`
   - Pass `onRecordingConsent` that calls `recordingState.submitConsent()`
6. In the in-call state:
   - Pass recording props to `VideoRoom` (which passes to `VideoControls`):
     - `isRecording={recordingState.isRecording}`
     - `onStopRecording={recordingState.stopRecording}`
     - `canStopRecording={isInterviewer}`
7. On `handleConnected` callback:
   - If recording_enabled and isInterviewer: call `recordingState.startRecording()` (auto-record per CONTEXT.md)
   - This triggers the Egress to begin when the call starts

**Candidate interview-client.tsx** -- Find this file. If it exists, integrate similarly but:
- Candidates are NOT interviewers, so `canStopRecording` is always false
- Consent is submitted with the magic link token: `recordingState.submitConsent(magicLinkToken)`
- No auto-start logic (only interviewer starts recording)
- If this file doesn't exist at the expected path, check `apps/candidate/` for the actual interview flow file and update accordingly

For both clients, the `VideoLobby` already received the `recordingEnabled` prop from Plan 04. The key addition here is:
- Wiring `useRecordingState` to manage the lifecycle
- Auto-starting recording on connect (portal interviewer only)
- Passing recording state to the VideoRoom/VideoControls

Keep changes focused on recording integration. Don't restructure existing code.
