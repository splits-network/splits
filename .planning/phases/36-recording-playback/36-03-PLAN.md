<task type="auto">
  <name>Task 1: Create recording API routes</name>
  <files>services/video-service/src/v2/interviews/recording-routes.ts</files>
  <action>
Create a new file `recording-routes.ts` (following the pattern of routes.ts being focused on interview CRUD, this file handles recording-specific endpoints).

Export `registerRecordingRoutes(app: FastifyInstance, config: RecordingRoutesConfig)` where config includes: `repository`, `recordingService`, `eventPublisher`.

Endpoints:

1. **POST /api/v2/interviews/:id/recording/start** (authenticated)
   - `requireUserContext(request)` to get clerkUserId
   - Fetch interview, verify the caller is the interviewer participant (not candidate)
   - Call `recordingService.startRecording(id)`
   - Return `{ data: { recording_status: 'recording' } }`
   - Error cases: interview not found (404), not interviewer (403), recording not enabled (400), already recording (400)

2. **POST /api/v2/interviews/:id/recording/stop** (authenticated)
   - `requireUserContext(request)` to get clerkUserId
   - Verify caller is interviewer
   - Call `recordingService.stopRecording(id)`
   - Return `{ data: { recording_status: 'processing' } }`
   - Error cases: not recording (400), not interviewer (403)

3. **POST /api/v2/interviews/recording/webhook** (NO AUTH -- LiveKit internal callback)
   - This receives egress status updates from LiveKit
   - Parse the webhook body: LiveKit sends `EgressInfo` protobuf or JSON with `egress_id`, `status`, `file_results`
   - Use `WebhookReceiver` from `livekit-server-sdk` to verify the webhook signature using the API key/secret
   - On status `EGRESS_COMPLETE`:
     - Extract file URL from `file_results[0].filename` (the Azure Blob path)
     - Extract duration from `file_results[0].duration` (nanoseconds, convert to seconds)
     - Extract file size from `file_results[0].size`
     - Call `recordingService.handleEgressComplete(egressId, blobUrl, durationSeconds, fileSize)`
     - Publish `interview.recording_ready` event via eventPublisher with: `{ interview_id, recording_url, duration_seconds, file_size_bytes }`
   - On status `EGRESS_FAILED`:
     - Call `recordingService.handleEgressFailed(egressId, error)`
   - Always return 200 (webhook receivers should not fail to prevent retry storms -- per Phase 35 decision)

4. **POST /api/v2/interviews/:id/recording/consent** (NO AUTH -- supports magic link candidates)
   - Accept `{ token?: string }` in body for candidate auth, OR use `requireUserContext` for authenticated users
   - If token provided: validate via tokenService, get participant_id
   - If no token: requireUserContext, look up participant by user_id
   - Call `repository.addRecordingConsent(interviewId, participantId)`
   - Return `{ data: { consented: true } }`

5. **GET /api/v2/interviews/:id/recording** (authenticated)
   - Return recording status and metadata for an interview
   - `{ data: { recording_enabled, recording_status, recording_duration_seconds, recording_blob_url (null unless authorized), recording_file_size_bytes } }`
   - Only return recording_blob_url if caller is a participant or company admin

Keep file under 200 lines. Use `requireUserContext` from `../shared/helpers` (same as existing routes).

</task>
