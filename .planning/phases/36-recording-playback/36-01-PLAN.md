```sql
ALTER TABLE interviews
    ADD COLUMN recording_enabled BOOLEAN NOT NULL DEFAULT false,
    ADD COLUMN recording_status TEXT DEFAULT NULL
        CHECK (recording_status IN ('pending', 'recording', 'processing', 'ready', 'failed')),
    ADD COLUMN recording_egress_id TEXT,
    ADD COLUMN recording_blob_url TEXT,
    ADD COLUMN recording_duration_seconds INT,
    ADD COLUMN recording_file_size_bytes BIGINT,
    ADD COLUMN recording_started_at TIMESTAMPTZ,
    ADD COLUMN recording_ended_at TIMESTAMPTZ,
    ADD COLUMN recording_consent_given_at TIMESTAMPTZ;
```

Add a partial index for finding recordings that need processing:
```sql
CREATE INDEX idx_interviews_recording_status
    ON interviews(recording_status)
    WHERE recording_status IS NOT NULL;
```

Also create a table for recording consent tracking per participant:
```sql
CREATE TABLE interview_recording_consents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    interview_id UUID NOT NULL REFERENCES interviews(id) ON DELETE CASCADE,
    participant_id UUID NOT NULL REFERENCES interview_participants(id) ON DELETE CASCADE,
    consented_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT uq_recording_consent UNIQUE (interview_id, participant_id)
);
```

**repository.ts** -- Add methods to the existing InterviewRepository class:
- `updateRecordingStatus(id, updates: { recording_status, recording_egress_id?, recording_blob_url?, recording_duration_seconds?, recording_file_size_bytes?, recording_started_at?, recording_ended_at? })` -- updates recording fields on interview
- `findByEgressId(egressId: string): Promise<Interview | null>` -- looks up interview by recording_egress_id
- `addRecordingConsent(interviewId, participantId): Promise<RecordingConsent>` -- inserts into interview_recording_consents
- `findRecordingConsents(interviewId): Promise<RecordingConsent[]>` -- fetches consents for interview

**recording-service.ts** -- New file, create `RecordingService` class:
- Constructor takes: `repository: InterviewRepository`, `livekitApiKey: string`, `livekitApiSecret: string`, `livekitHost: string` (ws URL of LiveKit server), `azureConfig: { accountName, accountKey, containerName }`
- Uses `EgressClient` from `livekit-server-sdk` (same package already used for AccessToken)
- `async startRecording(interviewId: string): Promise<void>`:
  1. Fetch interview by ID, validate status is 'scheduled' or 'in_progress' and recording_enabled is true
  2. Validate room_name exists
  3. Call `egressClient.startRoomCompositeEgress(roomName, { file: { filepath, output: { azure: { accountName, accountKey, containerName } } } })` with:
     - filepath: `recordings/{interview_id}/{interview_id}.mp4`
     - Resolution: 1280x720 at 30fps (720p composite)
     - Audio codec: opus, video codec: h264
  4. Store the returned egress_id via `repository.updateRecordingStatus(id, { recording_status: 'recording', recording_egress_id: egressInfo.egressId, recording_started_at: now })`
- `async stopRecording(interviewId: string): Promise<void>`:
  1. Fetch interview, validate recording_status is 'recording' and recording_egress_id exists
  2. Call `egressClient.stopEgress(egressId)`
  3. Update recording_status to 'processing', set recording_ended_at
- `async handleEgressComplete(egressId: string, fileUrl: string, duration: number, fileSize: number): Promise<string>`:
  1. Look up interview by egressId
  2. Update to recording_status: 'ready', set recording_blob_url, recording_duration_seconds, recording_file_size_bytes
  3. Return the interview_id (caller publishes the RabbitMQ event)
- `async handleEgressFailed(egressId: string, error: string): Promise<void>`:
  1. Look up interview, update recording_status to 'failed'

For the EgressClient instantiation, use `new EgressClient(livekitHost, livekitApiKey, livekitApiSecret)`. The livekit-server-sdk exports EgressClient.

Use `EncodedFileOutput` with `EncodedFileType.MP4` and `AzureBlobUpload` for the Azure config. The exact API from livekit-server-sdk:
```typescript
import { EgressClient, EncodedFileOutput, EncodedFileType, AzureBlobUpload } from 'livekit-server-sdk';
```

Keep the file under 200 lines. The service is focused on Egress lifecycle only -- no route handling.
