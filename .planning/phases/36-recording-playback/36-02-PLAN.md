<task type="auto">
  <name>Task 1: Update LiveKit Egress K8s deployment with Azure Blob config</name>
  <files>infra/k8s/livekit-egress/deployment.yaml</files>
  <action>
Update the existing `infra/k8s/livekit-egress/deployment.yaml` to add Azure Blob Storage configuration in the EGRESS_CONFIG_BODY. The egress service needs to know WHERE to write files.

However, the actual storage destination is specified per-request when calling the Egress API (not in the global config). The global egress config only needs the LiveKit connection info and temp storage. The Azure Blob credentials are passed per-request from the video-service's RecordingService.

So the deployment.yaml change is minimal -- just ensure the egress temp path has enough disk and the config is correct. Add a volume for temp storage to handle large recording files:

1. Add an `emptyDir` volume for `/tmp/egress` temp storage with a `sizeLimit: 10Gi`
2. Add the volumeMount to the container
3. Add environment variables for Azure Blob credentials from a new K8s secret `azure-blob-secrets`:
   - `AZURE_STORAGE_ACCOUNT_NAME` from secret key `account-name`
   - `AZURE_STORAGE_ACCOUNT_KEY` from secret key `account-key`
   - `AZURE_STORAGE_CONTAINER_NAME` from secret key `container-name`

These env vars are NOT used by egress itself (egress gets Azure config per-request), but are documented here as reference for what the K8s secret must contain. The video-service reads these to pass to each egress request.

Also add these same env vars to the video-service deployment (if it exists in infra/k8s) or document that they need to be added to the video-service environment.

Update the egress container resources -- egress does CPU-heavy encoding:
- requests: cpu 1000m (up from 500m), memory 1Gi (up from 512Mi)
- limits: cpu 4000m (keep), memory 8Gi (up from 4Gi for video encoding)
  </action>

  <done>Egress deployment has temp storage volume, increased resources for encoding workloads, and documentation of required Azure secrets.</done>

<task type="auto">
  <name>Task 2: Create Azure Blob lifecycle management policy</name>
  <files>infra/k8s/livekit-egress/azure-blob-lifecycle.json</files>
  <action>
Create a JSON lifecycle management policy file that can be applied to the Azure Blob Storage container via Azure CLI:

```bash
az storage management-policy create \
    --account-name $ACCOUNT_NAME \
    --policy @infra/k8s/livekit-egress/azure-blob-lifecycle.json
```

The policy JSON should contain:
1. Rule "move-to-cool": Move blobs in `recordings/` prefix to Cool tier after 30 days since modification
2. Rule "auto-delete": Delete blobs in `recordings/` prefix after 90 days since modification

Structure:
```json
{
  "rules": [
    {
      "enabled": true,
      "name": "move-recordings-to-cool",
      "type": "Lifecycle",
      "definition": {
        "actions": {
          "baseBlob": {
            "tierToCool": {
              "daysAfterModificationGreaterThan": 30
            }
          }
        },
        "filters": {
          "blobTypes": ["blockBlob"],
          "prefixMatch": ["recordings/"]
        }
      }
    },
    {
      "enabled": true,
      "name": "delete-old-recordings",
      "type": "Lifecycle",
      "definition": {
        "actions": {
          "baseBlob": {
            "delete": {
              "daysAfterModificationGreaterThan": 90
            }
          }
        },
        "filters": {
          "blobTypes": ["blockBlob"],
          "prefixMatch": ["recordings/"]
        }
      }
    }
  ]
}
```

Add a comment block at the top of the file (or a companion README) explaining:
- Apply with: `az storage management-policy create --account-name <name> --policy @infra/k8s/livekit-egress/azure-blob-lifecycle.json`
- Recordings live in Hot tier for 30 days (frequent playback), Cool tier 30-90 days, deleted at 90 days
- The 90-day retention matches CONTEXT.md decision
  </action>

  <done>Lifecycle policy JSON exists with move-to-cool at 30 days and delete at 90 days rules, scoped to recordings/ prefix.</done>
</task>
