  <action>
    Add a `getAdminStats(period: string)` method to each service's admin repository that returns:
    1. **Sparkline data**: An array of 10 data points representing counts per sub-period within the given period
    2. **Trend percentage**: Period-over-period change as a percentage (compare current period total vs previous period total)

    **Period mapping** (accept query param `period` as `7d|30d|90d|1y|all`):
    - `7d`: 7 daily buckets, compare vs prior 7 days
    - `30d`: 10 buckets of ~3 days each, compare vs prior 30 days
    - `90d`: 10 buckets of ~9 days each, compare vs prior 90 days
    - `1y`: 12 monthly buckets, compare vs prior year
    - `all`: 10 buckets evenly distributed over all data, no trend (trend=0)

    **Implementation approach** for time-series bucketing:
    Use Supabase queries with `created_at` filters. For each metric, run two queries:
    1. Current period: records where `created_at >= periodStart` grouped into N buckets
    2. Previous period: count of records where `created_at >= prevPeriodStart AND created_at < periodStart`

    For bucketing, fetch all records in the period with just `created_at` column (using `select('created_at')`), then bucket in JS code. This avoids needing raw SQL. Keep it simple — Supabase doesn't support GROUP BY date_trunc easily.

    Actually, a better approach: use `select('id', { count: 'exact', head: true })` with date range filters for each bucket. For 10 buckets that's 10 count queries + 1 for previous period = 11 queries per metric. Use `Promise.all` to parallelize.

    **Helper function** (add to each repository file or a shared local util):
    ```typescript
    function getPeriodConfig(period: string): { buckets: number; bucketMs: number; periodMs: number } {
      const DAY = 86_400_000;
      switch (period) {
        case '7d': return { buckets: 7, bucketMs: DAY, periodMs: 7 * DAY };
        case '90d': return { buckets: 10, bucketMs: 9 * DAY, periodMs: 90 * DAY };
        case '1y': return { buckets: 12, bucketMs: 30 * DAY, periodMs: 365 * DAY };
        case 'all': return { buckets: 10, bucketMs: 0, periodMs: 0 }; // special case
        default: return { buckets: 10, bucketMs: 3 * DAY, periodMs: 30 * DAY }; // 30d
      }
    }
    ```

    **identity-service** `getAdminStats(period)` returns:
    ```typescript
    {
      users: { sparkline: number[], trend: number, total: number },
    }
    ```
    - `sparkline`: array of user counts per bucket (users created in that time range)
    - `trend`: percentage change vs previous period (e.g., `12` means +12%)
    - `total`: current total count

    **ats-service** `getAdminStats(period)` returns:
    ```typescript
    {
      jobs: { sparkline: number[], trend: number, total: number },
      applications: { sparkline: number[], trend: number, total: number },
      applicationFunnel: { label: string, value: number }[],
    }
    ```
    - `applicationFunnel`: count of applications grouped by status (applied, screening, interview, offer, hired, rejected). Use `select('status')` then count in JS, or use separate count queries per status.

    **network-service** `getAdminStats(period)` returns:
    ```typescript
    {
      recruiters: { sparkline: number[], trend: number, total: number },
      recruiterStatus: { label: string, value: number }[],
    }
    ```
    - `recruiterStatus`: count of recruiters grouped by status (active, pending, suspended, inactive). Query: `select('id', { count: 'exact', head: true }).eq('status', statusValue)` for each status.

    **Route registration** — add to each service's routes.ts:
    ```typescript
    // GET /admin/stats?period=30d
    app.get('/admin/stats', async (request, reply) => {
      const { period } = request.query as { period?: string };
      const stats = await adminService.getAdminStats(period || '30d');
      reply.send({ data: stats });
    });
    ```

    **Service layer** — add `getAdminStats(period)` that delegates to repository.

    **Do NOT touch billing-service or notification-service** — the existing `/admin/counts` endpoints already provide the totals needed for those stat tiles (activeEscrow, pendingPayouts, activeNotifications). The frontend will combine stats from multiple services.

    **Keep files under 200 lines.** The repository files are currently ~120-210 lines. The `getAdminStats` method will add ~60-80 lines. If a repository would exceed 200 lines, extract the stats methods into a separate file like `stats-repository.ts` in the same directory and import it.
